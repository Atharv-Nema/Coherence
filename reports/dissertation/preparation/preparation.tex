\section{Actor Model in Pony} \label{sec:actor_model_pony}

The Pony programming language \cite{pony_language} is built following the actor model described in section \ref{sec:intro_motivation}. However, Pony introduces specific variations to the pure actor model to better support high-performance programming. This section explores Pony's actor model in detail.

\subsection{Basic syntax and semantics}
In Pony, an \textit{actor} is a template to create actor instances (just like a \textit{class} is a template to create \textit{objects}). When ambiguous, I use the terms \textit{actor template} and \textit{actor instance} instead of \textit{actor}. A Pony actor consists of three primary components:

\textbf{State:} This is the private data stored within the actor instance. This data cannot be accessed directly by any other actor instance.

\textbf{Constructors (\textsf{new}):} These are special functions used to initialize the state of an actor instance upon creation.

\textbf{Behaviours (\textsf{be}):} These are the asynchronous interface through which actors communicate. Each actor instance maintains a \textit{mailbox} for incoming messages. A behaviour invocation does not execute synchronously within the caller's execution context. Instead, the invocation is internally converted into a message and added to the recipient actor instance's mailbox for future processing. %CR: Perhaps a footnote about invocation vs calling

Let us consider a sample program to illustrate these ideas.

\begin{lstlisting}
actor Counter
  // Encapsulated private state
  var _count: U32
  // Constructor
  new create(init: int) =>
    _count = init
  // Behaviours
  be increment() =>
    _count = _count + 1
  be display(env: Env) =>
    env.out.print("Current count: " + _count.string())

actor Main
  new create(env: Env) =>
    let c = Counter.create(1) // actor instance creation
    // Invocation of behaviours
    c.increment()
    c.display(env)
\end{lstlisting}

This example demonstrates creating an actor instance and sending messages to it. Notably, the invocation \textsf{c.increment()} is non-blocking: the \textsf{Main} actor does not stall to wait for the counter to be updated. Despite this asynchronous nature, it is strictly guaranteed that the \textsf{display} behaviour invocation will be processed after the increment. This is ensured by \textit{causal ordering}, a property of the Pony mailbox system which guarantees that messages arrive in causal order (which in this case means the order they were invoked in).

\subsection{Mailbox and Runtime Characteristics}
Execution in Pony starts from the \textsf{create()} constructor of the \textsf{Main} actor and ends when all mailboxes are empty. The Pony runtime associates with each actor instance a mailbox, which is a FIFO queue of incoming messages. Whenever a behaviour is invoked, the runtime enqueues a message (containing data to identify the behaviour and its arguments) to the recipient actor instance's mailbox, and then returns control to the caller. The control returns only after the message has been added, which is what guarantees causal ordering. 

The Pony runtime maintains a pool of \textit{worker threads}, which are OS-level threads that execute actor behaviours. When an actor instance has pending messages, a worker thread is assigned to process and run the behaviour associated with the message. Messages of an actor instance are processed sequentially. This ensures that behaviours have data-race free access to the encapsulated state of the actor. The execution of a behaviour is never interrupted (runs to completion). This simplifies implementation as the runtime does not need a mechanism to save and restore the stack of the running behaviour as there is no context switching, and hence the behaviours can directly utilise the current worker thread's stack. However, this model is prone to hogging of worker threads. For this reason, behaviours in Pony are supposed to be short-lived.

\section{Reference capabilities} \label{sec:ref_cap_pony}
Reference capabilities are a novel concept introduced in Pony that enable shared-memory concurrency while statically preventing data races. They are a part of a reference's type and govern how the reference can be used. Each capability specifies both the operations the holder may perform (\textit{permissions}) and the restrictions imposed on other aliases (\textit{deny properties}). These deny properties are enforced at compile time, and reference capabilities carry no runtime overhead.

\subsection{Pony's memory model}
Unlike C or C++, Pony does not have raw pointers. It follows an object-oriented model similar to Java: data is encapsulated within classes, instances are heap-allocated, and access is via stack-stored references. Pony classes are not actors: they do not have mailboxes and behaviours, and serve purely to encapsulate state and functionality.

To illustrate this, consider the simple account class in Pony.

\begin{lstlisting}
class Account
  var balance: U64
  new create(opening_balance: U64) =>
    balance = opening_balance

actor Main
  new create(env: Env) =>
    // 'acc' is a reference on the stack pointing to 
    // an Account object on the heap. Each reference
    // comes with a capability, and acc's capability
    // is 'ref' (explained in detail later)
    let acc: Account ref = Account(100)
    // Members are accessed via the reference
    acc.balance += 50
\end{lstlisting}

\subsection{Specific capabilities used in Pony} \label{sec:preparation_pony_specific_capabilities}

The main capabilities used in Pony are \textsf{iso}, \textsf{ref}, \textsf{val}, and \textsf{tag}.

\begin{itemize}
    \item \textbf{\textsf{iso} (Isolated)}
    \begin{itemize}
        \item \textbf{Permissions:} Read and Write.
        \item \textbf{Deny Properties:} Denies both read and write access to all other aliases, both \textit{locally} (within the same actor instance) and \textit{globally} (across all actor instances).
        \item \textbf{Use Case:} An \textsf{iso} reference provides exclusive access to an object. Because the compiler guarantees that no other alias capable of reading or writing to the object exists, the holder can safely mutate the data without worrying about data races. An \textsf{iso} reference can be sent across actor boundaries, provided the sender \textit{consumes} the alias during the transfer. The compiler prevents the use of a consumed alias.
        \item \textbf{Code example:}
\begin{lstlisting}
class Data
    var value: String = "Secret"

actor Main
new create(env: Env) =>
    // Create an isolated object
    let d_iso: Data iso = Data
    d_iso.value = "Updated" // We can mutate it locally
    // To send it to another actor, we must consume it
    let other = OtherActor.create()
    other.receive_data(consume d_iso)
    // d_iso is now invalid. The following line would be a 
    // compile error:
    // env.out.print(d_iso.value)
\end{lstlisting}
    \end{itemize}
    
    
    \item \textbf{\textsf{ref} (Reference)}
    \begin{itemize}
        \item \textbf{Permissions:} Read and Write.
        \item \textbf{Deny Properties:} Denies nothing to local aliases, but denies all access to aliases in other actors.
        \item \textbf{Use Case:} This is the standard mutable reference for data confined to a single actor. Multiple \textsf{ref} aliases can exist and interact with the same object, but they are prohibited from crossing actor boundaries.
        \item \textbf{Code example:}
\begin{lstlisting}
let d_ref: Data ref = Data
// Can create multiple local aliases
let d_alias: Data ref = d_ref
d_alias.value = "Local Change" 
// Fails to compile because ref is not sendable:
// let other = OtherActor.create()
// other.receive_data(d_ref)
\end{lstlisting}
    \end{itemize}
    
    \item \textbf{\textsf{val} (Value)}
    \begin{itemize}
        \item \textbf{Permissions:} Read-only.
        \item \textbf{Deny Properties:} Denies write access to all other aliases globally.
        \item \textbf{Use Case:} Since a \textsf{val} object is guaranteed to be immutable, it can be shared with an unlimited number of actors simultaneously.
        \item \textbf{Code example:}
\begin{lstlisting}
let d_val: Data val = Data
// This would fail to compile as it is not modifiable
// d_val.value = "dummy"
// Can create local aliases
let other_val1 = d_val
let other_val2 = d_val
// Safe to share with other actors
let other_actor = OtherActor.create()
other_actor.receive_data(d_val)
\end{lstlisting}
    \end{itemize}

    \item \textbf{\textsf{tag} (Tag)}
    \begin{itemize}
        \item \textbf{Permissions:} None (can only be compared for equality).
        \item \textbf{Deny Properties:} Does not deny any aliases.
        \item \textbf{Use Case:} In Pony, the \textsf{tag} capability is primarily used for identifying objects or sending asynchronous messages. Since it allows neither reading nor writing, it can be shared globally without violating isolation. Its use case in \textit{Coherence} is less clear, primarily because \textit{Coherence} does not associate reference capabilities with actor instances. However, it is still needed for completeness for \textit{viewpoint adaptation}, which will be explored in the next section.
    \end{itemize}
\end{itemize}

\subsection{Viewpoint adaptation} \label{sec:preparation_viewpoint_adaptation}
Viewpoint adaptation is the mechanism Pony uses to correctly handle references to references.

Consider the following code:
\begin{lstlisting}
class A
  var x: U64 = 0

class B
  var a: A ref = A // Field 'a' is mutable within B

actor Main
  new create(env: Env) =>
    let b: B val = B // We have an immutable reference to B
\end{lstlisting}

What should the capability of \textsf{b.a} be? Since \textsf{b} has capability \textsf{val}, we are permitted to share it across actor boundaries. If \textsf{b.a} were allowed to retain its original \textsf{ref} capability, a recipient actor could gain mutable access to the object \textsf{a}. This would result in two different actors holding a \textsf{ref} to the same data simultaneously, directly violating \textsf{ref}'s deny properties and reintroducing data races. The solution to this problem is viewpoint adaptation. Whenever we access a field through a reference, we modify its type using the viewpoint adaptation operator (denoted by the symbol $\triangleright$). $\triangleright$ is a binary operator from $C^2 \rightarrow C$, where $C$ is the set of capabilities. Accessing a field with defined inner capability $F$ through an outer reference with capability $O$ gives a reference with capability $O \triangleright F$. So in the above example, \textsf{b.a} has type \textsf{A val} as $\textsf{val} \triangleright \textsf{ref} = \textsf{val}$.

The following table defines the result of the adaptation operator $O \triangleright F$ in Pony, where $O$ represents the capability of the originating reference and $F$ represents the capability of the field being accessed.

\section{Safe locking} \label{sec:safe_locking}
Pony safely reintroduces \textit{shared memory} using reference capabilities. This ensures data-race freedom at compile time. However, representing patterns where multiple actors frequently modify a shared data structure remains awkward without locks (see section \ref{sec:pony_expressive_power}). \textit{Coherence} aims to reintroduce locks while maintaining deadlock-freedom. This section describes some concepts needed for this.

\subsection{Locked reference capability}
The standard solution to shared memory is to protect the shared data structure with a lock, making sure that only one concurrent unit is accessing it at the same time. The Kappa type system by Elias Castegren\cite{kappa_system} integrates this idea into the reference capability model by introducing the \textsf{locked} reference capability. An object with this capability comes with an implicit lock. Whenever a method of the object is called, the lock associated with the object is taken, which is released after the method execution completes. This object can be shared between actors, and hence allows for easy sharing of mutable state. This fixes many of the shared data structure coordination issues that Pony faces. However, it is not perfect. As the implicit lock is acquired and released on each method call, operations involving multiple method calls or operations involving different shared data structures are not atomic. In addition, deadlocks remain possible when operations traverse object graphs containing multiple \textsf{locked} references, since different actors may acquire locks in different orders.

\subsection{Atomic sections and lock-set inference}
An atomic section is a programming construct that ensures a block of code executes as an indivisible, uninterrupted operation. In \textit{Coherence}, we want the programmers to be able to wrap code in atomic sections without any manual taking of locks. The compiler should automatically infer and acquire the necessary locks to guarantee data race and deadlock free execution.

\textit{Coherence} draws inspiration from Gudka work on automatic lock-inference for Java\cite{gudka_thesis}. Gudka's system \textit{Lockguard}  translates Java code containing atomic sections to an equivalent program that implements these atomic sections using locks. \textit{Lockguard} performs a lock-set inference on an atomic section to determine the locks needed, and then generates code to take these locks at the start of the atomic section and release them at the end.

\textit{Coherence} protects shared memory using an additional \textsf{locked<LOCK\_NAME>} reference capability. Unlike Kappa, programmers are expected to label the locks that protect a particular pointer (the locks are explicit). The idea is that a pointer can only be dereferenced once its associated lock has been acquired. However, \textit{Coherence} does not allow programmers to manually acquire locks (as they may write themselves into deadlocks). A pointer with a \textsf{locked} capability can only be dereferenced within an atomic section. Just like \textit{Lockguard}, the \textit{Coherence} compiler performs a lock-set inference to get the set of locks needed to protect all pointers in the section. It then acquires them in a canonical order to prevent deadlocks.

\section{Compilers} \label{sec:compiler_prep}
For the purpose of this project, a compiler consists of a frontend (lexer and parser), a validation phase, and a code generation phase. Languages typically also include a runtime, which is prewritten code linked with the compiled output to provide environmental support like memory management and scheduling. In \textit{Coherence}, the runtime is critical. It manages actor mailboxes, source-level locks and actor scheduling.

\subsection{Intermediate representation}
\textit{Coherence} compiles down to LLVM IR. LLVM IR offers several advantages:
\begin{itemize}
    \item \textbf{Architecture independence}: By targeting LLVM, the \textit{Coherence} compiler remains platform-agnostic (the lowering to machine-specific assembly is handled by the LLVM backend).
    \item \textbf{Out-of-box optimizations}: LLVM implements a vast amount of optimization passes. Compiling down to LLVM means that the \textit{Coherence} compiler gets these optimizations for free.
\end{itemize}


\subsection{Runtime} \label{sec:preparation_runtime}
The \textit{Coherence} runtime is implemented in C++20, chosen for its support for fine-grained concurrency primitives and C ABI compatibility, which enables seamless integration with LLVM. We spend the rest of this section talking about some design choices of the \textit{Coherence} runtime, and the necessary tools needed for them. %CR: rephrase this

Unlike Pony, \textit{Coherence} behaviours have the ability to pause execution before completion.

\begin{lstlisting}
be dummy_behaviour {
  var a: int locked<A>;
  atomic {
    *a; // Dereferencing [a] requires access to lock [A].
  }
}
\end{lstlisting}

In the code above, if \textsf{dummy\_behaviour} reaches the atomic section and the required lock \textsf{A} is held by another actor instance, \textsf{dummy\_behaviour} must wait until \textsf{A} is available. The runtime should not be idle until the lock is available. It should be able to move on to running other actor instances. When the lock is available again, the runtime should be able to resume the execution of the behaviour seamlessly.

One naive solution to this requirement is \textit{1:1 scheduling}. 1:1 scheduling is where each unit of concurrency (in \textit{Coherence}, this is the running behaviour), is mapped directly to its own OS-level thread. Under this model, utilizing standard OS-level mutexes to implement source-level locks would solve the suspension and resumption issues automatically, as the operating system would handle blocking the thread and resuming it once the lock becomes available. However, this approach has the following drawbacks:

\begin{itemize}
    \item Every suspension triggers a costly OS-level context switch. This creates a performance bottleneck when actors yield or block frequently.
    \item \textit{Coherence} aims for lightweight concurrency, where one can easily spawn thousands of actor instances without issues. Standard OS threads are ill-suited for this scale.
\end{itemize}

Hence, the \textit{Coherence} runtime uses \textit{M:N scheduling}, where a fixed pool of $N$ worker threads multiplexes the execution of $M$ actor behaviours. A critical consequence of this architecture is that behaviours cannot execute directly on the worker thread's native stack. If they did, suspending a behaviour would require copying its stack segment to preserve the execution context which is quite inefficient. The runtime must instead provide each behaviour with its own independent stack, and implement a mechanism for the worker thread to swap between different stacks. To implement this stack-swapping mechanism, the \textit{Coherence} runtime utilises the \textsf{boost} library's \textsf{fcontext\_t} type (\textsf{boost::context::detail::fcontext\_t}). This is a low-level, assembly-based primitive that facilitates highly efficient context switching entirely in user-space.

\subsection{Language choice for the compiler}
I chose C++20 as the primary language for developing the \textit{Coherence} compiler. The initial decision was driven by my familiarity with the language and its robust support for generating LLVM IR. However, I eventually decided not to use this API (see section \ref{sec:codegen}). In retrospect, a functional language like OCaml might have been better suited to this task. An unforeseen advantage of using C++ was that since the \textit{Coherence} runtime was written in C++, keeping the compiler in the same language allowed for a unified build system (CMake) and a shared testing infrastructure.

\section{Requirement analysis} \label{sec:requirement_analysis}
The development of \textit{Coherence} is guided by the following requirements.

\begin{itemize}
    \item \textbf{Static Safety}: Devise a type system and implement a type checker that strictly enforces data-race and deadlock freedom.
    \item \textbf{Compiler and Runtime}: Implement a working end-to-end compiler capable of lowering \textit{Coherence} source code to LLVM IR, and a runtime that satisfies the needs of the language.
    \item \textbf{Expressive power}: The language must be sufficiently expressive to allow programmers to implement diverse concurrent algorithms without the syntax becoming cumbersome.
\end{itemize}

\section{Software engineering techniques} \label{sec:software_techniques}
Various software tools and techniques were used to ensure that the project progressed smoothly.

\subsection{Build system}
I used \textit{CMake} as the project's build system. CMake has good integration with VSCode (my IDE) which allowed me to build my project and run tests with a single click.

\subsection{Version control system}
I utilised \textit{Git} for version control. Various implementation choices required significant trial and error. Using Git allowed me to explore new implementation ideas and refactor core logic with the safety net of being able to revert to previous stable states. To safeguard against hardware failure, the repository was hosted remotely on GitHub.

\subsection{Development model}
The development of Coherence followed the spiral model. This approach was essential for managing the technical risks inherent in building a custom type system and a low-level runtime. Instead of attempting to implement the full language specification at once, I proceeded through the following stages:

\begin{itemize}
    \item \textbf{Initial Cycle}: The first iteration focused on implementing the core of \textit{Coherence}. I implemented a restricted syntax that excluded high-level complexities such as lock-inference for mutually recursive functions or viewpoint adaptation for references to references. The goal of this phase was to establish a functional end-to-end compiler, ensuring that the skeleton of the system was operational. This provided a working foundation early in the project, allowing me to verify the core architecture before committing to the full implementation of more difficult language features.
    \item \textbf{Feature-Driven Iterations}: Once the core was stable, I performed subsequent cycles to iteratively expand the language's capabilities.
    \item \textbf{Incremental verification}: At the end of each cycle, I wrote sample programs using these new features and added unit and end-to-end tests to make sure that the features worked as intended.
\end{itemize}
