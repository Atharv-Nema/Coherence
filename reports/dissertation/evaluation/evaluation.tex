\chapter{Evaluation}
Following the project proposal, the evaluation of \textit{Coherence} focuses on correctness, expressive power and performance.

\section{Correctness}
There are a wide range of potential risks in this project. The compilation pipeline might contain bugs, the runtime could suffer from heisenbugs, and the type system may be unsound. This section discusses the methods used to mitigate these issues.

\subsection{Testing}
To detect bugs in the compilation pipeline, a test suite consisting of $62$ tests was developed. The largest portion of this suite consists of $47$ type-checking tests. They were designed to cover a wide range of programming constructs and target distinct components of the type-checking stage. The suite includes both positive and negative tests. Additionally, $4$ unit tests were created to verify the correct implementation of the lock-inference algorithm. The remaining $11$ tests perform end-to-end verification of the entire compilation pipeline (from parsing to codegen and linking with the runtime). Of these, $6$ are small, deterministic programs intended to confirm that core language constructs of \textit{Coherence} compile and run as expected. The other $5$ are concurrency-focused. These tests target potential runtime bugs. Most of these tests are large in scale, involving the creation of numerous actor instances and the transmission of many messages. This approach was necessary because heisenbugs were difficult to detect in short programs.

\subsection{Programming principles}
In addition to testing, to prevent bugs, the compiler extensively utilizes defensive programming. Assertions are integrated throughout the compilation pipeline to verify internal invariants. These checks act as an immediate signal for logic bugs.

Implementing the runtime correctly proved challenging due to frequent encounters with heisenbugs. To ensure correctness beyond standard testing, a structured programming paradigm was introduced. Shared runtime data structures were protected with locks to prevent data races. However, this alone was insufficient to guarantee correct behaviour.  Many operations needed to be performed atomically---for example, adding an actor instance to a lock's wait queue while simultaneously changing its state to \textsf{WAITING}. To address this, all such atomic operations were identified and wrapped in logical atomic sections. These sections were implemented by acquiring all necessary locks at the start of the section and releasing them at the end. To prevent deadlocks, the runtime was designed to always acquire locks in a fixed global order.


\subsection{Soundness of type system} \label{sec:type_system_soundness}
In the context of \textit{Coherence}, soundness is defined to be the absence of data races and deadlocks in programs which type check.

Deadlock in \textit{Coherence} is defined as a state where no actor instance is runnable or currently running, yet some actor instances have work left (their mailboxes are not empty). This can only occur when actor instances enter a cyclic wait within lock queues. This is prevented by \textit{Coherence} as the generated code takes locks in a globally fixed order, preventing cyclic dependencies. 

For data-race safety, we first consider \textit{Coherence1.0} which is much simpler as it does not have to deal with the complexities of viewpoint adaptation. Since \textsf{ref}, \textsf{val}, and \textsf{iso} are inherited directly from Pony, their data race freedom is already formally established\cite{pony_language}. The soundness argument for \textit{Coherence1.0} therefore focuses on the \textsf{locked<L>} capability and how it interacts with existing capabilities. The argument relies on two lemmas. The first states that if the deny properties of all reference capabilities hold (see section \ref{sec:preparation_pony_specific_capabilities}), memory accesses cannot result in a data race. The deny property of \textsf{locked<L>} requires that any read or write to the reference can only occur when the accessing actor holds lock \textsf{L}. Since mutual exclusion ensures that only one actor can hold \textsf{L} at a time, all accesses are synchronized, preventing data races. The second lemma states that all execution steps (aliasing, message passing, and so on) preserve these invariants. This holds because \textsf{locked<L>} can only be assigned to references of the same capability, and assignment from a different capability is only permitted when the source has no existing aliases. Together, these lemmas show that the reference capability invariants hold at every execution step, which implies safety throughout execution.

Establishing data-race safety for \textit{Coherence2.0} is considerably more complex. The interactions involving the \textsf{locked<L>} capability are more nuanced. For instance, we must now account for \textsf{locked<L>} appearing as a field type within nested pointer structures, requiring careful reasoning about aliasing across multiple levels of indirection. Additionally, as discussed in section \ref{sec:extension-pony-deviation}, the type checking rules of \textit{Coherence2.0} differ subtly from Pony. Fortunately, the assignability rules and the definitions of covariance and invariance presented in section \ref{sec:extension-typing-rules} are designed to ensure that assignments within a local scope preserve the invariants of reference capabilities, regardless of the concrete definition of the $\triangleright$ operator.

However, \textit{Coherence} has operations beyond simple local assignments. For these operations, the specific definition of $\triangleright$ plays a critical role in ensuring safety and must satisfy certain constraints. The remainder of this section discusses these constraints and how they are verified.

As mentioned in section \ref{sec:top-level-typing}, the \textit{Coherence} type system allows a behaviour to contain any argument whose outermost capability is not \textsf{ref}. However, an inappropriate definition of $\triangleright$ can undermine this simple check. Consider, for instance, if $\textsf{val} \triangleright \textsf{ref} = \textsf{ref}$. A behaviour could then accept an \textsf{(int ref) val} as an argument (as its outermost capability is not \textsf{ref}). This would allow actors to wrap a \textsf{ref} inside a \textsf{val} and send it to another actor, violating the invariant of \textsf{ref}. 

Similarly, the typing rules do not guarantee safety when unaliasing an \textsf{iso} for an ill-defined $\triangleright$. Suppose $\textsf{iso} \triangleright \textsf{ref} = \textsf{tag}$ and $\textsf{iso\textasciicircum} \triangleright \textsf{ref} = \textsf{ref}$. This would permit storing a \textsf{tag} to a \textsf{ref} field of an \textsf{iso}, then unaliasing to obtain a \textsf{ref} to what was originally a \textsf{tag}. This effectively allows assignment of a \textsf{tag} to a \textsf{ref}, which is unsafe. 

Finally, we must also verify that the definition of $\triangleright$ satisfies the nested consistency property mentioned at the end of section \ref{sec:extension-typing-rules}: when base types are identical, assignability, covariance, and invariance reduce to checking only the outer capabilities.

If the $\triangleright$ operator is associative, the above claims (which are quantified over all types with unbounded pointer nesting) reduce to checking only two levels of indirection (a pointer to a pointer to a base type). This makes exhaustive verification feasible. To this end, I wrote a Prolog program that verifies associativity of $\triangleright$, soundness of sendability, soundness of \textsf{iso} unaliasing, and nested consistency. This program can be found in appendix \ref{appendix:prolog}. The mechanical verification of these key non-trivial properties and intuitive confidence in the definition by reasoning through examples gives an assurance for the soundness of \textit{Coherence2.0}.

\section{Expressive power}
A core goal of \textit{Coherence} is to offer high expressive power without sacrificing safety. This section examines this claim. We begin by demonstrating how \textit{Coherence} handles coordinated access to shared data structures, comparing its approach with that of Pony. We then explore finer-grained coordination patterns, using the \textit{sleeping barber problem}\cite{sleeping_barber} as a case study. Following this, section \ref{sec:memory_model} evaluates the memory model of \textit{Coherence} in terms of safety and flexibility. Finally, we discuss the limitations of \textit{Coherence}.

\subsection{Shared data structures} \label{sec:pony_expressive_power}
I argue that \textit{Coherence} is better suited than lock-free languages like Pony for expressing concurrent operations on shared data structures. 

In Pony, shared mutable state is managed by passing exclusive ownership between actors via \textsf{iso} references. When multiple actors need access to a shared resource, a common approach (which I call the \textit{hot-potato pattern}) uses a central scheduler actor to coordinate access. Actors request the resource from the scheduler. If the resource is not available, the scheduler puts the requesting actor in a wait queue. When the resource becomes available, the scheduler transfers ownership of the resource to the requesting actor. After completing its operation, the actor returns the resource to the scheduler.

This pattern works adequately for operations on a single shared resource. However, difficulties arise when an operation must atomically operate on multiple shared resources. Consider the problem of atomically swapping the values of two shared integers. Conceptually, the swapping actor must first request the first integer, wait to receive it, request the second integer, wait again, perform the swap and finally return the integers. Since Pony behaviours cannot block, this operation must be fragmented across multiple behaviours, with intermediate state stored in actor fields. A complete implementation is provided in appendix \ref{appendix:hot-potato}.

This pattern is also quite error-prone. Programmers must manually return resources. A forgotten return permanently loses the resource with no compile-time safeguard. This pattern is also susceptible to logical deadlocks: states that are technically not deadlocks from the language's perspective but exhibit identical symptoms from the programmer's viewpoint. Consider two actors (\textsf{A} and \textsf{B}) attempting to access the same two schedulers (\textsf{s1} and \textsf{s2}), but in a different order:
\begin{itemize}
  \item \textsf{A} acquires the resource from \textsf{s1}, and \textsf{B} from \textsf{s2}.
  \item \textsf{A} requests \textsf{s2} and enters its wait queue. \textsf{B} requests \textsf{s1} and enters its wait queue.
\end{itemize}

Each actor waits for the other to return the resource. Neither can proceed, and the system is stuck. Since Pony terminates when all mailboxes are empty, this program exits prematurely. The core issue is that although the runtime mailboxes of the actor instances are empty, there is still work to be done from the perspective of the programmer (the wait queues of the scheduler actors are not empty). Compilers cannot be expected to catch such bugs as they cannot interpret programmer intent encoded in programmer-defined structures. However, expressive languages should eliminate the need for such error-prone patterns altogether.

\textit{Coherence} solves this through the \textsf{locked<L>} capability. Shared data is protected by associating it with a lock, and operations spanning multiple shared structures are wrapped in an \textsf{atomic} block. As shown below, swapping two shared integers reduces to a straightforward sequential program. Locks are acquired and released automatically and safely, eliminating the need for patterns prone to logical deadlocks.

\begin{lstlisting}
actor Swapper
  be swap(ptr1: int locked<A>, ptr2: int locked<B>) =>
    atomic {
      // The compiler infers that both lock A and lock B 
      // are required. They are acquired in a fixed order
      // (e.g., A then B) before the code below executes.
      let temp = *ptr1
      *ptr1 = *ptr2
      *ptr2 = temp
    }
    // Both locks are automatically released here.
\end{lstlisting}



\subsection{Sleeping and Waking via message passing}
Complex concurrent algorithms often require finer-grained coordination beyond basic mutual exclusion. Consider the \textit{sleeping barber problem}\cite{sleeping_barber}, which involves a barber who sleeps when no customers are present and customers who must wake a sleeping barber upon arrival, provided there is space in the waiting room. Unlike simple locking where an entity acquires and releases a lock for mutual exclusion, this problem requires the ability to sleep until woken by another entity. A standard solution is as follows:

\begin{algorithmic}[1]
\Procedure{Barber}{\textsf{shop}}
    \While{\textbf{true}}
        \State \textbf{acquire} \textsf{shop.mutex}
        
        \If{\textsf{shop.queue} is empty}
            \State \text{// Atomically release \textit{shop.mutex} and sleep until signalled}
            \State \textbf{wait\_on} \textsf{shop.cond\_var} with \textsf{shop.mutex}
            \State \text{// Condition variable ensures that \textit{shop.mutex} is}
            \State \text{// implicitly re-acquired upon waking up }
        \EndIf
        
        \State $\textsf{customer} \gets \textsf{shop.queue.pop\_front()}$
        \State \textbf{release} \textsf{shop.mutex}
        
        \State \Call{PerformHaircut}{\textsf{customer}s}
    \EndWhile
\EndProcedure

\medskip
\hrule
\medskip

\Procedure{CustomerVisit}{$\textsf{shop}, \textsf{customer}$}
    \State \textbf{acquire} \textsf{shop.mutex}
    
    \If{$\textsf{shop.queue.size} < \textsf{shop.max\_chairs}$}
        \State \textsf{shop.queue.push\_back(customer)}
        \State \text{// Notify the barber that a customer is ready}
        \State \textbf{signal} \textsf{shop.cond\_var}
        \State \textbf{release} \textsf{shop.mutex}
    \Else
        \State \text{// Shop is full. Customer leaves}
        \State \textbf{release} \textsf{shop.mutex}
        \State \Return
    \EndIf
\EndProcedure
\end{algorithmic}

The solution uses a mutex for data-race free access of \textsf{shop.queue} and a condition variable to sleep until a customer arrives.

\textit{Coherence} does not provide direct access to synchronization primitives like semaphores or condition variables. However, equivalent patterns can be simulated through message passing combined with shared state. The key insight is that shared data structures and actor fields persist across behaviour invocations. To sleep, an actor simply allows its current behaviour to complete. All necessary state is retained in actor members. To wake up, another actor sends a message that triggers a new behaviour invocation, resuming operation. Using this insight, we can solve the \textit{sleeping barber problem} as follows.

\begin{lstlisting}
actor Barber {
  // A FIFO queue storing customer information
  queue: customer_queue locked<L>;
  new create(int num_chairs) {
    queue := create_queue(num_chairs);
  }

  be wake_up() {
    var work_remaining: bool = true;
    // Initialize work_remaining
    atomic {
      work_remaining = !queue_empty(queue);
    }
    // Process all waiting customers, then implicitly "sleep"
    // by allowing the behaviour to complete
    while (work_remaining) {
        atomic {
            var c: Customer = pop_customer(queue);
            c.finish_haircut();
            work_remaining = !queue_empty(queue);
        }
    }
    // Barber "sleeps" until the next wake_up message
  }
}

actor Customer {
  be visit_shop(Barber barber, (customer_queue locked<L>) queue) {
    atomic {
      if(!queue_full(queue)) {
        add_customer(queue, this);
        // Will still work without the check below. However,
        // many useless wake_up messages will be sent.
        if(!queue_empty(queue)) {
          barber.wake_up(); // Wake the barber
        }
      }
      // Otherwise, leave (no space in the waiting room)
    }
  }
}
\end{lstlisting}

When a customer arrives, it atomically checks for space in \textsf{queue}, enqueues itself if space exists, and sends a \textsf{wake\_up} message to the barber if the queue is empty. The barber processes all queued customers within its behaviour. When the queue empties, the behaviour completes and the barber effectively "sleeps" until another \textsf{wake\_up} arrives.

The logical correctness of the wake-up protocol remains the programmer's responsibility. However, the \textsf{atomic} sections guarantee safe concurrent access to the shared queue, preventing data races and ensuring consistent state. Unlike the standard imperative solution, no explicit mutex management is required. The compiler handles synchronization automatically, allowing the programmer to focus on the coordination logic itself.

\subsection{Memory model} \label{sec:memory_model}
We evaluate \textit{Coherence}'s memory model in terms of safety and expressive power.

Regarding safety, \textit{Coherence} is not memory safe: it permits null pointers and does not perform bounds checking on array accesses. This contrasts with Pony, which guarantees memory safety by ensuring all pointers are initialized and performs runtime bounds checking. The null pointer---famously described as "the billion dollar mistake" by its inventor Tony Hoare\cite{hoare_null}---is a well-known source of bugs. \textit{Coherence} could get rid of null pointers by introducing variants, requiring no changes to the core type system. However, fixing unsafe array accesses would require runtime bounds checking, conflicting with \textit{Coherence}'s principle of zero runtime overhead. Whether this modest performance cost is worth the safety guarantees remains a design trade-off.

Regarding expressive power, the \textit{Coherence} memory model offers greater flexibility than Pony through its support for \textit{value semantics}. Structures in \textit{Coherence} are true values that can be stored directly on the stack or inlined within other structures, enabling copy-on-assignment and cache-friendly memory layouts without pointer indirection. Pony, by contrast, is built on reference semantics: user-defined structures are stored on the heap and accessed via references. This precludes the flexible stack allocation and explicit copying patterns available in \textit{Coherence}.

\subsection{Limitations of \textit{Coherence}} \label{sec:coherence-limitations}
We analyse the limitations from the perspective of concurrent programming and general code style.

\paragraph{Concurrent programming} \textit{Coherence}'s type system cannot capture lock-free programming patterns (for example, compare and swap). While lock-free operations are not more expressive (any such operation can be simulated by acquiring a lock, performing the operation, and then releasing the lock), they are significantly more efficient in practice. Additionally, the \textsf{locked<L>} capability lacks the granularity to support multiple-reader single-writer patterns. Another major limitation of \textit{Coherence} is that the number of locks is fixed at compile time, with each protected object tied to a statically known lock type. Consequently, patterns such as a shared linked list where each node is protected by a distinct lock cannot be expressed:
\begin{lstlisting}
type linked_list = struct {
  value: int;
  // All nodes share lock L. No way to vary locks per node
  next: linked_list locked<L>; 
}
\end{lstlisting}
Type systems exist that guarantee deadlock freedom while allowing for dynamic creation of locks at runtime\cite{dynamic_locks}. Integrating such a system into \textit{Coherence} was planned as an extension. However, it proved more complex than anticipated, requiring substantial changes to the language, and was ultimately abandoned.

\paragraph{General code style} \textit{Coherence} lacks first-class functions, making callback-based patterns difficult to express directly. It also lacks parametric polymorphism over types, so common abstractions like a generic list cannot be implemented. This limitation is exacerbated by the lock system: data protected by different locks constitute distinct types, and without the ability to abstract over types, a function operating on \textsf{unit locked<A>} for example, cannot accept \textsf{unit locked<B>}.


\section{Performance}
This section briefly examines some performance characteristics of \textit{Coherence}, examining whether the performance-oriented design choices discussed throughout this dissertation deliver their intended benefits.
\subsection{Compiler performance}
As discussed in section \ref{sec:time-complexity-analysis}, the compiler guarantees a worst-case time complexity of $\Theta(N^2)$, where $N$ represents the size of the input program. This bound ensures that compilation times remain predictable and manageable for typical programs. Achieving this complexity required devising an optimized lock-inference algorithm, as discussed in section \ref{sec:optimised-inference}.

To empirically verify this bound, we measure the compilation time of the adversarial program $\Phi_n$ (described in section \ref{sec:time-complexity-analysis}). Figure \ref{fig:compilation_time} plots compilation time against $n$, which serves as a proxy for the size of $\Phi_n$. The results confirm that the observed time complexity is quadratic, consistent with our theoretical analysis. It is worth noting that $\Phi_n$ is a specifically constructed adversial program: typical program's compilation times are expected to be closer to linear.

\subsection{Runtime performance}
A key design decision in the \textit{Coherence} runtime was to employ $M:N$ scheduling with user-level context switching, rather than dedicating an OS thread to each actor instance (see section \ref{sec:preparation_runtime}). To empirically evaluate this choice, I implemented the ring-token benchmark, a standard test in which a token is passed $m$ times around a ring of $n$ actors. I configured the benchmark with $m = 10^6$ passes and $n = 10^3$ actors, chosen to stress-test the runtime with frequent context switches and high message-passing volume.

I compared three implementations: \textit{Coherence}, Pony, and a C++ implementation meant to simulate a thread-per-actor runtime implementation. The C++ implementation arranges $1000$ OS threads in a ring and uses semaphores to coordinate token passing. This implementation understates the true cost of a thread-per-actor design, as it omits the additional overhead of supporting message queues and message passing that a complete implementation would require.

The results, shown in figure~\ref{fig:ring-token}, demonstrate that \textit{Coherence} (4.49s) outperforms the C++ implementation (4.98s) despite the latter's simplified implementation. This supports the claim that efficiently managing thousands of actor instances requires lighter-weight abstractions than OS threads.

However, \textit{Coherence} is slower than Pony (2.97s), which is expected for two reasons. First, Pony is a more mature language that benefits from various of runtime optimization. Second, Pony's runtime behaviours directly on worker thread stacks. In contrast, the \textit{Coherence} runtime must allocate and deallocate a separate stack for each behaviour invocation and perform explicit user-space context switches as explained in section \ref{sec:preparation_runtime}, introducing additional overhead.

\subsection{Methodology}
All benchmarks were executed on a machine equipped with an Intel Core Ultra 7 155H processor (3.80 GHz, 22 cores) and 32 GB RAM. Each measurement represents the average of five runs. When comparing \textit{Coherence} and Pony, both runtimes were configured to use the same number of worker threads.
