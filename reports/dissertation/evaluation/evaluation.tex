\chapter{Evaluation}
Following the project proposal, \textit{Coherence}'s evaluation focuses on correctness, expressive power and performance.

\section{Correctness}
This project faces several risks: compilation pipeline bugs, runtime heisenbugs, and type system unsoundness. This section discusses the methods used to mitigate these issues.

\subsection{Testing}
To detect compilation pipeline bugs, a test suite containing $62$ tests was developed. The majority of this suite consists of $47$ type-checking tests. These were designed to cover a wide range of programming constructs and target distinct components of the type-checking stage. The suite includes both positive and negative tests. Additionally, $4$ unit tests were created to verify the correct implementation of the lock-inference algorithm. The remaining $11$ tests perform end-to-end verification of the entire compilation pipeline (from parsing to codegen and linking with the runtime). Of these, $6$ are small, deterministic programs intended to confirm \textit{Coherence}'s core language constructs compile and run as expected. The other $5$ are concurrency-focused. These tests target potential runtime bugs. Most of these tests are large in scale, involving creating numerous actor instances and transmitting many messages. This approach was necessary because heisenbugs were difficult to detect in short programs.

\subsection{Programming principles}
In addition to testing, to prevent bugs, the compiler extensively utilizes defensive programming. Assertions are integrated throughout the compilation pipeline to verify internal invariants. These checks act as an immediate signal for logic bugs.

Implementing the runtime correctly proved challenging due to frequent encounters with heisenbugs. To ensure correctness beyond standard testing, a structured programming paradigm was introduced. Shared runtime data structures were protected with locks to prevent data races. However, this alone was insufficient to guarantee correct behaviour.  Many operations needed to be performed atomically---for example, adding an actor instance to a lock's wait queue while simultaneously changing its state to \texttt{WAITING}. To address this, all such atomic operations were identified and wrapped in logical atomic sections. These sections were implemented by acquiring all necessary locks at the start of the section and releasing them at the end. To prevent deadlocks, the runtime was designed to always acquire locks in a fixed global order. %cite


\subsection{Soundness of type system} \label{sec:type_system_soundness}
\textit{Coherence} defines soundness as data-races and deadlocks freedom in programs which type check.

Deadlock in \textit{Coherence} is defined as a state where no actor instance is runnable or currently running, yet some actor instances have work left (their mailboxes are not empty). This can only occur when actor instances enter a cyclic wait within lock queues. This is prevented by \textit{Coherence} as the generated code takes locks in a globally fixed order, preventing cyclic dependencies. 

For data-race safety, we first consider \textit{Coherence1.0}. Since \texttt{ref}, \texttt{val}, and \texttt{iso} are inherited directly from Pony, their data race freedom is already formally established\cite{pony_language}. The soundness argument for \textit{Coherence1.0} therefore focuses on the \texttt{locked<L>} capability and how it interacts with existing capabilities. The argument relies on two lemmas. 

\begin{description}
  \item[Safety:] If the deny-properties of all reference capabilities hold (see section \ref{sec:preparation_pony_specific_capabilities}), memory accesses cannot result in a data race. 
  \item[Preservation:] All execution steps (aliasing, message passing, unaliasing and so on) preserve the deny-properties of references.
\end{description}

For showing safety, the deny-property of \texttt{locked<L>} requires that any read or write to the reference can only occur when the accessing actor holds lock \texttt{L}. Mutual exclusion ensures only one actor holds \texttt{L} at a time, preventing data races. For showing preservation, \texttt{locked<L>} can only be assigned to references of the same capability, and assignment from a different capability is only permitted when the source has no existing aliases. Together, these lemmas show that the deny-properties hold at every execution step, which implies safety throughout execution.

Establishing data-race safety for \textit{Coherence2.0} is considerably more complex. The interactions involving the \texttt{locked<L>} capability are more nuanced. For instance, we must now account for \texttt{locked<L>} appearing as a field type within nested pointer structures, requiring careful reasoning about aliasing across multiple levels of indirection. Additionally, as discussed in section \ref{sec:extension-pony-deviation}, the type checking rules of \textit{Coherence2.0} differ subtly from Pony. Fortunately, the assignability rules presented in section \ref{sec:extension-typing-rules} are designed to ensure that assignments within a local scope preserve the invariants of reference capabilities, regardless of the concrete definition of the $\triangleright$ operator.

However, \textit{Coherence} has operations beyond simple local assignments. For these operations, the specific definition of $\triangleright$ plays a critical role in ensuring safety and must satisfy certain constraints. 
\begin{description}
  \item[Sendability safety:] As mentioned in section \ref{sec:top-level-typing}, the \textit{Coherence} type system allows a behaviour to contain any argument whose outermost capability is not \texttt{ref}. However, an inappropriate definition of $\triangleright$ can undermine this simple check. Suppose we defined $\texttt{val} \triangleright \texttt{ref} = \texttt{ref}$. A behaviour could accept an \texttt{(int ref) val} as an argument (as its outermost capability is not \texttt{ref}). This would allow actors to wrap a \texttt{ref} inside a \texttt{val} and send it to another actor, violating the invariant of \texttt{ref}.
  \item[Unaliasing safety:] An ill-defined $\triangleright$ can make unaliasing an \texttt{iso} unsafe. Suppose $\texttt{iso} \triangleright \texttt{ref} = \texttt{ref}$ and $\texttt{iso\textasciicircum} \triangleright \texttt{ref} = \texttt{iso\textasciicircum}$. This would permit storing a \texttt{ref} to a \texttt{ref} field of an \texttt{iso}, then unaliasing to obtain a \texttt{iso\textasciicircum}, unsafely upgrading a potentially aliased \texttt{ref} into an \texttt{iso\textasciicircum}.
  \item[Nested consistency:] $\triangleright$ must satisfy the nested consistency property mentioned in section \ref{sec:extension-typing-rules}: when base types are identical, assignability and compatibility reduce to checking only the outer capabilities.
\end{description}

Although these safety claims are quantified over types with unbounded pointer nesting, the recursive definition of the type-checking rules ensures that they can be verified by considering only two levels of indirection (a pointer to a pointer to a base type). This makes exhaustive verification feasible. To this end, I wrote a Prolog program that verifies sendability safety, unaliasing safety, and nested consistency. This program can be found in appendix \ref{appendix:prolog}. Mechanically verifying these key non-trivial properties and intuitive confidence in the definition by reasoning through examples assures us of the soundness of \textit{Coherence2.0}.

\section{Expressive power}
A core goal of \textit{Coherence} is to offer high expressive power without sacrificing safety. This section examines this claim.

\subsection{Shared data structures} \label{sec:pony_expressive_power}
I argue that \textit{Coherence} is better suited than lock-free languages like Pony for expressing concurrent operations on shared data structures. 

Pony programs share mutable state by passing exclusive ownership over the state using \texttt{iso} references. When multiple actors need access to a shared resource, a common approach (which I call the \textit{hot-potato pattern}) uses a central scheduler actor to coordinate access. Actors request the resource from the scheduler. If the resource is not available, the scheduler puts the requesting actor in a wait queue. When the resource becomes available, the scheduler transfers ownership of the resource to the requesting actor. After completing its operation, the actor returns the resource to the scheduler.

This pattern works adequately for operations on a single shared resource. However, difficulties arise when an operation must atomically operate on multiple shared resources. Consider the problem of atomically swapping the values of two shared integers. Conceptually, the swapping actor must first request the first integer, wait to receive it, request the second integer, wait again, perform the swap and finally return the integers. Since Pony behaviours cannot block, this operation must be fragmented across multiple behaviours, with intermediate state stored in actor fields. Appendix \ref{appendix:hot-potato} provides the complete implementation.

This pattern is also quite error-prone. Programmers must manually return resources. A forgotten return permanently loses the resource with no compile-time safeguard. This pattern is also susceptible to logical deadlocks: states that are technically not deadlocks from the language's perspective but exhibit identical symptoms from the programmer's viewpoint. Consider two actors (\texttt{A} and \texttt{B}) attempting to access the same two schedulers (\texttt{s1} and \texttt{s2}), but in a different order:
\begin{itemize}
  \item \texttt{A} acquires the resource from \texttt{s1}, and \texttt{B} from \texttt{s2}.
  \item \texttt{A} requests \texttt{s2} and enters its wait queue. \texttt{B} requests \texttt{s1} and enters its wait queue.
\end{itemize}

Each actor waits for the other to return the resource. Neither can proceed, and the system is stuck. Since Pony terminates when all mailboxes are empty, this program exits prematurely. The core issue is that although the runtime mailboxes are empty, there is still work to be done from the perspective of the programmer (the user-defined wait queues of the scheduler actors are not empty). Compilers cannot be expected to catch such bugs as they cannot interpret programmer intent encoded in user-defined structures. However, expressive languages should eliminate the need for such error-prone patterns altogether.

\textit{Coherence} solves this through the \texttt{locked<L>} capability. Shared data is protected by associating it with a lock, and operations spanning multiple shared structures are wrapped in an \texttt{atomic} block. As shown below, swapping two shared integers reduces to a straightforward sequential program. Locks are acquired and released automatically and safely, eliminating the need for patterns prone to logical deadlocks.

\begin{minted}{pony}
actor Swapper {
  be swap(ptr1: int locked<A>, ptr2: int locked<B>) {
    atomic {
      var temp: int = *ptr1
      *ptr1 = *ptr2
      *ptr2 = temp
    }
  }
}
\end{minted}



\subsection{Sleeping and Waking via message passing}
Complex concurrent algorithms often require finer-grained coordination beyond basic mutual exclusion. Consider the \textit{sleeping barber problem}\cite{sleeping_barber}, which involves a barber who sleeps when no customers are present and customers who must wake a sleeping barber upon arrival, provided there is space in the waiting room. Unlike simple locking where an entity acquires and releases a lock for mutual exclusion, this problem requires the ability to sleep until woken by another entity. A standard solution is as follows:

\begin{algorithmic}[1]
\Procedure{Barber}{\texttt{shop}}
    \While{\textbf{true}}
        \State \textbf{acquire} \texttt{shop.mutex}
        
        \If{\texttt{shop.queue} is empty}
            \State \text{// Atomically release \textit{shop.mutex} and sleep until signalled}
            \State \textbf{wait\_on} \texttt{shop.cond\_var} with \texttt{shop.mutex}
            \State \text{// Condition variable ensures that \textit{shop.mutex} is}
            \State \text{// implicitly re-acquired upon waking up }
        \EndIf
        
        \State $\texttt{customer} \gets \texttt{shop.queue.pop\_front()}$
        \State \textbf{release} \texttt{shop.mutex}
        
        \State \Call{PerformHaircut}{\texttt{customer}s}
    \EndWhile
\EndProcedure

\medskip
\hrule
\medskip

\Procedure{CustomerVisit}{$\texttt{shop}, \texttt{customer}$}
    \State \textbf{acquire} \texttt{shop.mutex}
    
    \If{$\texttt{shop.queue.size} < \texttt{shop.max\_chairs}$}
        \State \texttt{shop.queue.push\_back(customer)}
        \State \text{// Notify the barber that a customer is ready}
        \State \textbf{signal} \texttt{shop.cond\_var}
    \EndIf
    \State \textbf{release} \texttt{shop.mutex}
\EndProcedure
\end{algorithmic}

The solution uses a mutex for data-race free access to \texttt{shop.queue} and a condition variable to sleep until a customer arrives.

\textit{Coherence} does not provide direct access to synchronization primitives like semaphores or condition variables. However, equivalent patterns can be simulated through message passing combined with shared state. The key insight is that shared data structures and actor fields persist across behaviour invocations. To sleep, an actor simply allows its current behaviour to complete. All necessary state is retained in actor members. To wake up, another actor sends a message that triggers a new behaviour invocation, resuming operation. Using this insight, we can solve the \textit{sleeping barber problem} as follows.

\begin{minted}{Pony}
actor Barber {
  // A FIFO queue storing customer information
  queue: customer_queue locked<L>;
  new create((customer_queue locked<L>) queue_arg) {
    queue := queue_arg;
  }

  be wake_up() {
    // Process all waiting customers, then implicitly "sleep"
    // by allowing the behaviour to complete
    while (!queue_empty(queue)) {
        atomic {
            var c: Customer = pop_customer(queue);
            c.perform_haircut();
        }
    }
    // Barber "sleeps" until the next wake_up message
  }
}

actor Customer {
  be visit_shop(Barber barber, (customer_queue locked<L>) queue) {
    atomic {
      if(!queue_full(queue)) {
        add_customer(queue, this);
        // Will still work without the check below. However,
        // many useless wake_up messages will be sent.
        if(!queue_empty(queue)) {
          barber.wake_up(); // Wake the barber
        }
      }
      // Otherwise, leave (no space in the waiting room)
    }
  }
}
\end{minted}

When a customer arrives, it atomically checks for space in \texttt{customer\_queue}, enqueues itself if space exists, and sends a \texttt{wake\_up} message to the barber if the queue is empty. The barber processes all queued customers within its behaviour. When the queue empties, the behaviour completes and the barber effectively "sleeps" until another \texttt{wake\_up} arrives.

The logical correctness of the wake-up protocol remains the programmer's responsibility. However, the \texttt{atomic} sections guarantee safe concurrent access to the shared queue, preventing data races and ensuring consistent state. Unlike the standard imperative solution, no explicit mutex management is required. The compiler handles synchronization automatically, allowing the programmer to focus on the coordination logic itself.

\subsection{Memory model} \label{sec:memory_model}
We evaluate \textit{Coherence}'s memory model in terms of safety and expressive power.

Regarding safety, \textit{Coherence} is not memory safe: it permits null pointers and does not perform bounds checking on array accesses. This contrasts with Pony, which guarantees memory safety by ensuring all pointers are initialized and performs runtime bounds checking. The null pointer---famously described as "the billion dollar mistake" by its inventor Tony Hoare\cite{hoare_null}---is a well-known source of bugs. \textit{Coherence} could get rid of null pointers by introducing variants, requiring no changes to the core type system (cf. Java's move towards option). However, fixing unsafe array accesses would require runtime bounds checking, conflicting with \textit{Coherence}'s principle of zero runtime overhead. Whether this modest performance cost is worth the safety guarantees remains a design trade-off.
%cite: confer with java or something
Regarding expressive power, the \textit{Coherence} memory model offers greater flexibility than Pony through its support for \textit{value semantics}. Structures in \textit{Coherence} are true values that can be stored directly on the stack or inlined within other structures, enabling copy-on-assignment and cache-friendly memory layouts without pointer indirection. Pony, by contrast, is built on \textit{reference semantics}: user-defined structures are stored on the heap and accessed via references. This precludes the flexible stack allocation and explicit copying patterns available in \textit{Coherence}.

\subsection{Limitations of \textit{Coherence}} \label{sec:coherence-limitations}
We analyse the limitations from the perspective of concurrent programming and general code style.

\paragraph{Concurrent programming} \textit{Coherence}'s type system cannot capture lock-free programming patterns (for example, compare and swap). While lock-free operations are not more expressive (any such operation can be simulated by acquiring a lock, performing the operation, and then releasing the lock), they are significantly more efficient in practice. Additionally, the \texttt{locked<L>} capability lacks the granularity to support multiple-reader single-writer patterns. Another major limitation of \textit{Coherence} is that the number of locks is fixed at compile time, with each protected reference tied to a statically known lock. Consequently, patterns such as a shared linked list where each node is protected by a distinct lock cannot be expressed:
\begin{minted}[escapeinside=||]{text}
|\textcolor{mygreen}{\textbf{type}}| node = struct {
  value: int;
  // All nodes share lock L. No way to vary locks per node
  next: node locked<L>; 
}
\end{minted}
Type systems exist that guarantee deadlock freedom while allowing dynamic creation of locks at runtime\cite{dynamic_locks}. Integrating such a system into \textit{Coherence} was planned as an extension. However, it proved more complex than anticipated, requiring substantial changes to the language, and was ultimately abandoned.

\paragraph{General code style} \textit{Coherence} lacks first-class functions, making callback-based patterns difficult to express directly. It also lacks parametric polymorphism over types, so common abstractions like a generic list cannot be implemented. This limitation is exacerbated by the lock system: data protected by different locks constitute distinct types, and without the ability to abstract over types, a function operating on \texttt{unit locked<A>} for example, cannot accept \texttt{unit locked<B>}.


\section{Performance}
This section briefly examines some performance characteristics of \textit{Coherence}, examining whether the performance-oriented design choices discussed throughout this dissertation deliver their intended benefits.
\subsection{Compiler performance}
As discussed in section \ref{sec:time-complexity-analysis}, the compiler guarantees a worst-case time complexity of $\Theta(N^2)$, where $N$ is the input program size. This bound ensures that compilation times remain predictable and manageable for typical programs. Achieving this complexity required devising an optimized lock-inference algorithm, as discussed in section \ref{sec:optimised-inference}.

To empirically verify this bound, we measure the compilation time of the adversarial program $\Phi_n$ (described in section \ref{sec:time-complexity-analysis}). Figure \ref{fig:compilation_time} plots compilation time against $n$, which serves as a proxy for the size of $\Phi_n$. The results confirm that the observed time complexity is quadratic, consistent with our theoretical analysis. It is worth noting that $\Phi_n$ is a specifically constructed adversial program: typical program's compilation times are expected to be closer to linear.

\subsection{Runtime performance}
A key design decision in the \textit{Coherence} runtime was to employ $M:N$ scheduling with user-level context switching, rather than dedicating an OS thread to each actor instance (see section \ref{sec:preparation_runtime}). To empirically evaluate this choice, I implemented the many-to-many ping-pong benchmark. In this test, $n$ actor instances are created and each is tasked with initiating $m$ request-response exchanges with peers selected randomly. I configured the benchmark with $m = 100$ exchanges and $n = 1000$ actors, chosen to stress-test the runtime with numerous actor instances, frequent context switches and high message-passing volume.

I compared three implementations: \textit{Coherence}, Pony, and a C++ implementation meant to simulate a thread-per-behaviour runtime implementation. The results, shown in \Cref{fig:ping-pong}, demonstrate that \textit{Coherence} (0.63s) greatly outperforms the C++ implementation (6.20s). This supports the claim that efficiently managing thousands of actor instances requires lighter-weight abstractions than OS threads.

However, \textit{Coherence} is significantly slower than Pony (0.19s), which is expected for two reasons. Firstly, Pony is a more mature language that benefits from various runtime optimization. Secondly, Pony's runtime executes behaviours directly on worker thread stacks. In contrast, as \textit{Coherence} behaviours might suspend mid-execution, the \textit{Coherence} runtime must allocate and deallocate a separate stack for each behaviour invocation and perform explicit user-space context switches (see section \ref{sec:preparation_runtime}), introducing additional overhead.

\subsection{Methodology}
All benchmarks were executed on a machine equipped with an Intel Core Ultra 7 155H processor (3.80 GHz, 22 cores) and 32 GB RAM. Each measurement represents the mean of five runs, with error bars indicating one sample standard deviation. When comparing \textit{Coherence} and Pony, both runtimes were configured to use the same number of worker threads.
