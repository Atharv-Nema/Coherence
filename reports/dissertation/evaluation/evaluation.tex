\chapter{Evaluation}
Following the project proposal, the evaluation of \textit{Coherence} focuses on correctness, expressive power and performance.

\section{Correctness}
There are a wide range of potential inaccuracies in this project. The compilation pipeline might contain bugs, the runtime could suffer from heisenbugs, and the type system may be unsound. This section discusses the methods used to mitigate these issues.

\subsection{Testing}
% (92 words)
To detect bugs in the compilation pipeline implementation, a test suite consisting of 62 tests was developed. The largest portion of this suite consists of 47 type-checking tests. They were designed to cover a wide range of programming constructs and target distinct components of the type-checking stage. The suite includes both positive and negative tests. Additionally, 4 unit tests were created to verify the correct implementation of lock-inference algorithm. The remaining 11 tests perform end-to-end verification of the entire compilation pipeline (from parsing to codegen and linking with the runtime). Of these, six are small, deterministic programs intended to confirm that core language constructs of \textit{Coherence} compile and run as expected. The other five are concurrency-focused. These tests target potential runtime bugs. Most of these tests are large in scale, involving the creation of numerous actor instances and the transmission of many messages. This approach was necessary because heisenbugs were difficult to detect in short programs, and many bugs only became apparent when the tests were scaled up.


\subsection{Correctness of type-system}
% 647 words
In the context of \textit{Coherence}, soundness is defined to be the absence of data races and deadlocks.

Deadlock in \textit{Coherence} is defined as a state where no actor instance is runnable or currently running, yet some actor instances have work left (their mailboxes are not empty). This can only occur when actor instances enter a cyclic wait within lock queues. This is prevented by \textit{Coherence} as the generated code takes locks in a globally fixed order, preventing cyclic dependencies. 

For data race safety, we first address \textit{Coherence1.0} which is much simpler as it does not have to deal with the complexities of viewpoint adaptation. Since \textsf{ref}, \textsf{val}, and \textsf{iso} are inherited directly from Pony, their data race freedom is already formally established. The soundness argument for \textit{Coherence1.0} therefore focuses on the \textsf{locked<L>} capability and how it interacts with existing capabilities. The argument relies on two lemmas. The first states that if all reference capability invariants hold, memory accesses cannot result in a data race. The invariant of\textsf{locked<L>} requires that any read or write to the reference can only occur when the accessing actor holds lock \textsf{L}. Since mutual exclusion ensures only one actor can hold \textsf{L} at a time, all accesses are synchronized, preventing data races. The second lemma states that all execution steps (aliasing, message passing, and so on) preserve these invariants. This holds because \textsf{locked<L>} can only alias with references of the same capability, and assignment from a different capability is only permitted when the source has no existing aliases. Together, these lemmas show that invariants hold at every execution step, and that invariant preservation implies safety throughout execution.

Establishing data-race safety for \textit{Coherence 2.0} is considerably more complex. The interactions involving the \textsf{locked<L>} capability are more nuanced. For instance, we must now account for \textsf{locked<L>} appearing as a field type within nested pointer structures, requiring careful reasoning about aliasing across multiple levels of indirection. Additionally, as discussed in section \ref{sec:extension-pony-deviation}, the type checking rules of \textit{Coherence2.0} differ subtly from Pony. Fortunately, the assignability rules and the definitions of covariance and invariance presented in section \ref{sec:extension-typing-rules} are designed to ensure that assignments within a local scope preserve the invariants of any reference capability, regardless of the concrete definition of the $\triangleright$ operator.

However, \textit{Coherence} programs have operations beyond simple local assignments such as sending references to other actors and unaliasing \textsf{iso} references. The specific definition of $\triangleright$ plays a critical role in these operations and must satisfy certain constraints.

As mentioned in section (...), the \textit{Coherence} type-system allows a behaviour to contain any argument whose outermost capability is not \textsf{ref}. However, an inappropriate definition of $\triangleright$ can undermine this simple check. Consider, for instance, if $\text{val} \triangleright \text{ref} = \text{ref}$. A behaviour could then accept an \textsf{(int ref) val} as an argument (as its outermost capability is not \textsf{ref}). This would allow actors to wrap a \textsf{ref} inside a \textsf{val} and send it to another actor, violating the invariant of \textsf{ref}. 

Similarly, the type-checker do not guarantee safety when unaliasing an \textsf{iso} for an ill-defined $\triangleright$. Suppose $\text{iso} \triangleright \text{ref} = \text{tag}$ while $\text{iso\textasciicircum} \ \triangleright \text{ref} = \text{ref}$. This would permit storing a \textsf{tag} to a \textsf{ref} field of an \textsf{iso}, then unaliasing to obtain a \textsf{ref} to what was originally a \textsf{tag}. This effectively allows assignment of a \textsf{tag} to a \textsf{ref}, which is unsafe. 


Finally, we must also verify that the definition of $\triangleright$ satisfies the property mentioned in section \ref{...}: when base types are identical, assignability, covariance, and invariance reduce to checking only the outer capabilities.

If the $\triangleright$ operator is associative, the above claims (which are quantified over all types with unbounded pointer nesting) reduce to checking only two levels of indirection (a pointer to a pointer to a base type). This makes exhaustive verification feasible. To this end, I wrote a Prolog program that verifies associativity of $\triangleright$, soundness of sendability, soundness of \textsf{iso} unaliasing, and the property claimed in section \ref{...}. This program can be found in section (...). While these arguments are not completely rigorous, exhaustively verifying that my definition of $\triangleright$ satisfied these key non-trivial properties, and intuitive confidence in the definition by informally reasoning through examples gives an assurance for soundness of \textit{Coherence2.0}.

\section{Expressive power}
A core goal of \textit{Coherence} is to offer high expressive power without sacrificing safety. This section examines this claim. We begin by demonstrating how \textit{Coherence} handles coordinated access to shared data structures, comparing its approach with that of Pony. We then explore finer-grained coordination patterns, using the sleeping barber problem as a case study. Following this, we evaluate the memory model in terms of safety and flexibility. Finally, we discuss the limitations of the \textit{Coherence} language.

\subsection{Shared data structures}
I argue that \textit{Coherence} is better suited than lock-free languages like Pony for expressing concurrent operations on shared data structures. The main pattern in Pony to handle operations across multiple shared data structures is what I call the hot-potato pattern. In this pattern, a requesting actor receives exclusive ownership of the shared data structure via an \textsf{iso} reference. After completing the required operation, the holding actor passes the data to the next actor needing access, typically through a central scheduler. 
\begin{notebox}
The code presented below is intentionally simplified and not syntactically precise Pony, as full accuracy would require substantial boilerplate that distracts from the core concepts. Syntactically correct versions are provided in appendix (...). The code also makes use of first-class functions, which Pony supports with appropriate safety checks for cross-actor invocation. Since \textit{Coherence} does not support first-class functions, these checks are not discussed further. Here, first-class functions serve merely as a mechanism to abstract over behaviours, which is necessary because Pony does not permit behaviours themselves to be passed as values.
\end{notebox}

The following code demonstrates how the hot-potato pattern can be used to manage a shared integer:

\begin{lstlisting}
// A simple shared container to hold our value
class IntContainer
  var value: U64 = 0
  new create(v: U64) => value = v

actor IntScheduler
  var _data: (IntContainer iso | None)
  // A queue containing pending callbacks
  var _wait_queue: Deque[{(IntContainer iso): None} val]

  // Receives a first-class function
  be take(cb: {(IntContainer iso): None} val) =>
    match _data
    | let p: IntContainer iso => cb(consume p)
    | None => _wait_queue.push(cb)
    end

  be return(p: IntContainer iso) =>
    if _wait_queue.size() > 0 then
      let cb = _wait_queue.shift()
      cb(consume p)
    else 
      _data = consume p
    end
\end{lstlisting}

When an actor calls \textsf{take}, it passes a callback that will be invoked with the data once available. If the data is currently held by the scheduler, the callback is invoked immediately. Otherwise, it is queued. When \textsf{return} is called, the scheduler either dispatches the data to the next waiting callback or retains it for future requests.

Now, we consider the problem of performing operations across multiple different shared data structures. For example, consider atomically swapping the values of two shared integers. We can use the hot-potato model to solve this:

\begin{lstlisting}
actor Swapper
  var s1: IntScheduler
  var s2: IntScheduler
  var p1: (IntContainer iso | None) = None

  be receive2(p2: IntContainer iso) =>
    // Step 3: Perform the swap
    let temp = p1.value
    p1.value = p2.value
    p2.value = temp
    // Step 4: Return both
    s1.give(consume p1)
    s2.give(consume p2)

  be receive1(p1_arg: IntContainer iso) =>
    p1 = consume p1_arg
    // Step 2: Request the second shared integer
    s2.take({(p2: IntContainer iso): None =>
      this.receive2(consume p2)})

  be swap() =>
    // Step 1: Request the first shared integer
    s1.take({(p1: IntContainer iso): None =>
      self.receive1(consume p1)})
\end{lstlisting}

To perform a swap, the actor first requests access from the first scheduler. Upon receiving the data, it requests access from the second scheduler, and finally performs the swap once it holds both references exclusively. This example illustrates several issues with the hot-potato pattern. Since Pony behaviours cannot block, what is conceptually a single operation must be split across multiple behaviours, with intermediate state stored in actor fields. Also, this pattern is quite prone to bugs. The system relies entirely on programmers to return the resource. A forgotten return permanently loses the resource. More critically, this pattern can lead to logical deadlocks: states that are not technically deadlocks from the language's perspective, but exhibit identical symptoms from the programmer's viewpoint. Consider two actors attempting to access the same two schedulers, but in a different order:
\begin{itemize}
  \item Actor A acquires \textsf{p1} from \textsf{s1}, then calls \textsf{s2.take()} and enters \textsf{s2}'s wait queue.
  \item Actor B acquires \textsf{p2} from \textsf{s2}, then calls \textsf{s1.take()} and enters \textsf{s1}'s wait queue.
\end{itemize}
Each actor waits for the other to return. Neither can proceed, and the system is stuck. Since Pony terminates when all mailboxes are empty, this program exits prematurely. The core issue is that although the runtime mailboxes of the actor instances are empty, there is still work to be done from the perspective of the programmer (the wait queue of \textsf{IntScheduler} is not empty). Compilers cannot be expected to catch such bugs as they cannot interpret programmer intent encoded in user-defined structures. However, expressive languages should eliminate the need for such error-prone patterns altogether.

\textit{Coherence} solves this through the \textsf{locked<L>} capability. Shared data is protected by associating it with a lock, and operations spanning multiple structures are wrapped in an \textsf{atomic} block. The compiler infers the required locks and acquires them in a fixed global order, guaranteeing deadlock freedom.

\begin{lstlisting}
actor Swapper
  be swap(ptr1: int locked<A>, ptr2: int locked<B>) =>
    // The compiler infers that both lock A and lock B 
    // are required. They are acquired in a fixed order
    // (e.g., A then B) before the code below executes.
    atomic {
      let temp = *ptr1
      *ptr1 = *ptr2
      *ptr2 = temp
    }
    // Both locks are automatically released here.
\end{lstlisting}

As shown, swapping two shared integers reduces to straightforward sequential code without fragmented code, manual \textsf{take}/\textsf{return} or user-managed queues. Locks are acquired and released automatically, eliminating the need for patterns prone to logical deadlocks.

\subsection{Finer-grained coordination patterns}
Complex concurrent algorithms often require finer-grained coordination beyond basic mutual exclusion. Consider the \textit{sleeping barber problem} (cite), which involves a barber who sleeps when no customers are present and customers who must wake a sleeping barber upon arrival, provided there is space in the waiting room. Unlike simple locking where an entity acquires and releases a lock for mutual exclusion, this problem requires the ability to sleep until woken by another entity. A standard pseudocode solution is as follows:

\begin{algorithmic}[1]
\Procedure{Barber}{$shop$}
    \While{\textbf{true}}
        \State \textbf{acquire} $shop.mutex$
        
        \If{$shop.queue$ is empty}
            \State \text{// Atomically release \textit{shop.mutex} and sleep until signalled}
            \State \textbf{wait\_on} $shop.cond\_var$ with $shop.mutex$
            \State \text{// Condition variable ensures that \textit{shop.mutex} is}
            \State \text{// implicitly re-acquired upon waking up }
        \EndIf
        
        \State $customer \gets shop.queue.pop\_front()$
        \State \textbf{release} $shop.mutex$
        
        \State \Call{PerformHaircut}{$customer$}
    \EndWhile
\EndProcedure

\medskip
\hrule
\medskip

\Procedure{CustomerVisit}{$shop, customer$}
    \State \textbf{acquire} $shop.mutex$
    
    \If{$shop.queue.size < shop.max\_chairs$}
        \State $shop.queue.push\_back(customer)$
        \State \text{// Notify the barber that a customer is ready}
        \State \textbf{signal} $shop.cond\_var$
        \State \textbf{release} $shop.mutex$
    \Else
        \State \text{// Shop is full. Customer leaves}
        \State \textbf{release} $shop.mutex$
        \State \Return
    \EndIf
\EndProcedure
\end{algorithmic}

The solution uses a mutex for data-race free access of \textit{shop.queue}, and a condition variable to sleep until a customer arrives. A condition variable, rather than a simple semaphore, is required because the barber must atomically release and re-acquire \textit{shop.mutex}.

\textit{Coherence} does not provide direct access to synchronization primitives like semaphores or condition variables. However, equivalent patterns can be simulated through message passing combined with shared state. The key insight is that shared data structures and actor fields persist across behaviour invocations. To sleep, an actor simply allows its current behaviour to complete. All necessary state is retained in shared memory or actor members. To wake up, another actor sends a message that triggers a new behaviour invocation, resuming operation. Using this insight, we can solve the sleeping barber's problem as follows.

\begin{lstlisting}
actor Barber {
  // A FIFO queue storing customer information
  queue: customer_queue locked<L>;
  new create(int num_chairs) {
    queue := create_queue(num_chairs);
  }

  be wake_up() {
    // Process all waiting customers, then implicitly "sleep"
    // by allowing the behaviour to complete
    while (!queue_empty(queue)) {
        atomic {
            var c: Customer = pop_customer(queue);
            c.finish_haircut();
        }
    }
    // Barber "sleeps" until the next wake_up message
  }
}

actor Customer {
  be visit_shop(Barber barber, (customer_queue locked<L>) queue) {
    atomic {
      if(!queue_full(queue)) {
        add_customer(queue, this);
        // Will still work without the check below. However,
        // many useless wake_up messages will be sent.
        if(!queue_empty(queue)) {
          barber.wake_up(); // Wake the barber
        }
      }
      // Otherwise, leave (no space in waiting room)
    }
  }
}
\end{lstlisting}

When a customer arrives, it atomically checks for space in \textsf{queue}, enqueues itself if space exists, and sends a \textsf{wake\_up} message to the barber if the queue was empty. The barber processes all queued customers within its behaviour. When the queue empties, the behaviour completes and the barber effectively "sleeps" until another \textsf{wake\_up} arrives.

The logical correctness of the wake-up protocol remains the programmer's responsibility. \textit{Coherence} cannot verify that \textsf{wake\_up} is called at appropriate times, and that \textsf{wake\_up} is handled correctly. However, the \textsf{atomic} sections guarantee safe concurrent access to the shared queue, preventing data races and ensuring consistent state. Unlike the standard imperative solution, no explicit mutex management is required. The compiler handles synchronization automatically, allowing the programmer to focus on the coordination logic itself.

\subsection{Memory model}
We evaluate \textit{Coherence}'s memory model in terms of safety and expressive power.

Regarding safety, \textit{Coherence} is not memory safe: it permits null pointers and does not perform bounds checking on array accesses. This contrasts with Pony, which guarantees memory safety by ensuring all pointers are initialized and performing runtime bounds checking. For a production language, memory safety would be essential—the null pointer, famously called the "billion-dollar mistake" by its inventor Tony Hoare (cite ...), is a well-known source of bugs. The null issue could be addressed by introducing variant types, requiring no changes to the core type system. However, fixing unsafe array accesses would require runtime bounds checking, conflicting with \textit{Coherence}'s principle of zero runtime overhead. Whether this modest performance cost is worth the safety guarantees remains a design trade-off.

Regarding expressive power, the \textit{Coherence} memory model offers greater flexibility than Pony through its support for value semantics. Structures in \textit{Coherence} are true values that can be stored directly on the stack or inlined within other structures, enabling copy-on-assignment and cache-friendly memory layouts without pointer indirection. Pony, by contrast, is built on reference semantics: user-defined structures are stored on the heap and accessed via references. This precludes the flexible stack allocation and explicit copying patterns available in \textit{Coherence}.

\subsection{Limitations of \textit{Coherence}}
We analyse the limitations from the perspective of concurrent programming and general code style.

For concurrency, \textit{Coherence}'s type system cannot capture lock-free programming patterns that remain data-race safe. While lock-free operations are not strictly more expressive (any such operation can be simulated by acquiring a lock, performing the operation, and then releasing the lock), they are significantly more efficient in practice. Additionally, the \textsf{locked<L>} capability lacks the granularity to support multiple-reader-single-writer patterns. A major limitation of \textit{Coherence} is that the number of locks is fixed at compile time, with each protected object tied to a statically-known lock type. Consequently, patterns such as a shared linked list where each node is protected by a distinct lock cannot be expressed:
\begin{lstlisting}
type linked_list = struct {
  value: int;
  // All nodes share lock L; no way to vary locks per node
  next: linked_list locked<L>; 
}
\end{lstlisting}
Type systems exist that guarantee deadlock freedom while permitting unbounded locks at runtime (cite ...). Integrating such a system into \textit{Coherence} was considered as an extension. However, it proved more complex than anticipated, requiring substantial changes to the language design, and was ultimately abandoned.

For general code style, \textit{Coherence} lacks first-class functions, making callback-based patterns like the hot-potato model discussed in section (...) difficult to express directly. It also lacks parametric polymorphism over types, so common abstractions like a generic list cannot be implemented. This limitation is exacerbated by the lock system: data protected by different locks constitutes distinct types, and without the ability to abstract over lock parameters, a function operating on \textsf{unit locked<A>} for example cannot accept \textsf{unit locked<B>}.


\section{Performance}
This section briefly examines some of the performance characteristics of \textit{Coherence}, examining whether the performance-oriented design choices discussed throughout in this dissertation deliver their intended benefits.
\subsection{Compiler performance}
As discussed in section (...), the compiler guarantees a worst-case time complexity of $\Theta(N^2)$, where $N$ represents the size of the input program. This bound ensures that compilation times remain predictable and manageable for typical programs. Achieving this complexity required coming up with a new lock-inference algorithm, as discussed in section (...).

To empirically verify this bound, we measure the compilation time of the adversarial program $\Phi_n$ (described in section ...). This program consists of $n$ functions arranged to maximize the work performed by the lock-inference algorithm. Figure \ref{fig:compilation_time} plots compilation time against $n$, which serves as a proxy for code size. The results confirm that the observed time complexity is quadratic, consistent with our theoretical analysis.

\subsection{Runtime performance}
A key design decision in the \textit{Coherence} runtime was to employ $M:N$ scheduling with user-level context switching, rather than dedicating an OS thread to each actor instance (see section ...). To empirically evaluate this choice, we implemented the ring-token benchmark, a standard test in which a token is passed $m$ times around a ring of $n$ actors. We configured the benchmark with $m = 10^6$ passes and $n = 10^3$ actors, chosen to stress-test the runtime with frequent context switches and high message-passing volume.

We compared three implementations: \textit{Coherence}, Pony, and a C++ implementation meant to simulate a thread-per-actor runtime implementation. The C++ implementation arranges 1000 OS threads in a ring and uses semaphores to coordinate token passing. This implementation understates the true cost of a thread-per-actor design, as it omits the additional overhead of supporting message queues and message passing that a complete implementation would require.

The results, shown in figure~\ref{fig:ring-token}, demonstrate that \textit{Coherence} (4.49s) outperforms the C++ implementation (4.98s) despite the latter's simplified implementation. This supports the claim that efficiently managing thousands of actor instances requires lighter-weight abstractions than OS threads.

However, \textit{Coherence} is slower than Pony (2.97s), which is expected for two reasons. First, Pony is a more mature language that benefits a lot of runtime optimization (cite ...). Second, Pony's actor model guarantees that behaviours never block mid-execution, allowing the runtime to execute actor code directly on worker thread stacks. In contrast, \textit{Coherence} must allocate and deallocate a separate stack for each behaviour invocation and perform explicit user-space context switches, introducing additional overhead.

\subsection{Methodology}
All benchmarks were executed on a machine equipped with an Intel Core Ultra 7 155H processor (3.80 GHz, 22 cores) and 32 GB RAM. Each measurement represents the average of five runs. When comparing \textit{Coherence} and Pony, both runtimes were configured to use an identical number of worker threads.
